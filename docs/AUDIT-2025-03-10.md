## Placeholder Pages — Frontend Data Connectivity Audit

An additional audit of all beta/advanced pages to determine whether they are genuinely connected to real Kubernetes data. Five pages were found to be **effectively placeholders**: they call real backend analyzer functions but pass empty data, so they always return empty results.

### Root Cause

All 5 placeholder pages share the same architectural pattern: the Go backend exposes **pure analyzer functions** that accept raw Kubernetes data as arguments. This is a clean separation, but the frontend never fetches the real data to pass in — it sends empty objects or arrays.

### Placeholder Pages

#### P1 — SecurityOverview.tsx

- **Frontend call**: `CheckPodSecurity({})` — passes an empty object
- **Backend**: `CheckPodSecurity(podSpec map[string]any)` in `internal/security/scanner.go` — iterates `podSpec["containers"]` which is `nil`
- **Result**: Always shows "No security violations detected"
- **Fix needed**: Fetch all pods via `useKubeResources`, then pass each pod's spec to the scanner

#### P2 — NetworkPolicyGraph.tsx

- **Frontend call**: `BuildNetworkGraph([], [])` — passes empty arrays for policies and pods
- **Backend**: `BuildGraph(policies, pods []map[string]any)` in `internal/netpol/builder.go` — builds groups and edges from the input
- **Result**: Always shows an empty graph with 0 edges
- **Fix needed**: Fetch network policies and pods via `useKubeResources`, then pass the raw items to the builder

#### P3 — RBACGraph.tsx

- **Frontend call**: `BuildRBACGraph([], [], [], [])` — passes four empty arrays
- **Backend**: `BuildGraph(roles, clusterRoles, bindings, clusterBindings []map[string]any)` in `internal/rbacgraph/builder.go`
- **Result**: Always shows "0 bindings"
- **Fix needed**: Fetch roles, clusterroles, rolebindings, and clusterrolebindings via `useKubeResources`, then pass raw items to the builder

#### P4 — GitOps.tsx

- **Frontend call**: `DetectProviders([])` — passes an empty API groups list
- **Backend**: `Detect(apiGroups []string)` in `internal/gitops/detector.go` — checks for `argoproj.io`, `source.toolkit.fluxcd.io`, etc.
- **Result**: Always shows "No GitOps providers detected"
- **Fix needed**: Fetch server API groups (e.g. via a new backend handler or existing discovery endpoint) and pass them in

#### P5 — TroubleshootingPanel.tsx

- **Frontend call**: `Investigate(kind, namespace, name, {})` — passes an empty `{}` for the status parameter
- **Backend**: `diagnose(inv, status)` in `internal/troubleshoot/engine.go` — checks `status["reason"]`, `status["phase"]`, `status["exitCode"]` — all `nil` from the empty map
- **Result**: Always says "No known issues detected" (unless timeline database has entries)
- **Fix needed**: After the user enters a resource name, fetch the resource's actual status (e.g. pod `containerStatuses`) and pass it to `Investigate`
